# AGENTS.md

Rules for AI coding agents (Cursor, Claude Code, etc.) working in this repo.

## Project context

Bias Dungeon is a static browser game about cognitive biases, built with React + Vite + TypeScript + Tailwind + Framer Motion + Howler.js, deployed as a static site to Vercel. The authoritative source docs are:

- Product requirements: [`ai_docs/core_docs/prd.md`](ai_docs/core_docs/prd.md)
- Architecture design: [`ai_docs/core_docs/add.md`](ai_docs/core_docs/add.md)
- Work breakdown: [`ai_docs/core_docs/wbs.md`](ai_docs/core_docs/wbs.md)
- Live decision scratchpad: [`ai_docs/scratchpad/add_scratchpad.md`](ai_docs/scratchpad/add_scratchpad.md)

Read these before proposing changes that touch architecture, data shape, or scope.

## Git and GitHub conventions

The repository owner is the human, not the agent. Follow these rules without exception:

- **Use `gh` for all GitHub-facing operations** (repo creation, remote setup, pull requests, releases). This keeps authentication tied to the owner's GitHub account.
- **Never push code.** Do not run `git push`, `gh repo sync`, or any equivalent. Stop at the local commit and let the owner push.
- **Never create remotes or repos on the owner's behalf** unless they explicitly ask in the current turn. Prefer proposing the `gh` command and letting them run it.
- **Do not modify git identity.** Never run `git config user.name`, `git config user.email`, or any `git config --global`. The owner's existing config already attributes commits correctly.
- **Local `git add` and `git commit` are allowed** when the owner explicitly asks for a commit. Do not commit speculatively.

## Commit messages

Follow **Conventional Commits** format:

```
type(scope): short description
```

Allowed types: `feat`, `fix`, `chore`, `refactor`, `docs`, `style`, `test`, `perf`, `ci`, `build`.

Examples:
- `feat(rooms): add anchoring bias scenario`
- `fix(audio): resume sound after companion dialog`
- `chore(deps): bump framer-motion to 11.x`
- `refactor(state): simplify room progression logic`

Rules:
- Keep the subject under 72 characters.
- Use the body only when the diff needs context that the code doesn't convey.
- **Never add agent attribution.** No `Co-authored-by: Claude ...`, no `Generated with Cursor`, no "made by" trailers, no emoji signatures, no tool watermarks in the body or footer.

## Code style

- No comments that narrate what the code does. Comments are only for non-obvious intent, trade-offs, or constraints.
- No "generated by" or "created with" comments in any file.
- Match existing formatting (Prettier defaults + the ESLint config in [`eslint.config.js`](eslint.config.js)).

## Asking vs. doing

- If a decision would change scope, cost, or which service is used (hosting, analytics, auth, AI provider), ask before acting.
- If the task is ambiguous, ask one or two pointed questions instead of guessing.
- Log locked decisions in [`ai_docs/scratchpad/add_scratchpad.md`](ai_docs/scratchpad/add_scratchpad.md).

## Out of scope for MVP

Per the ADD §10 and WBS, the following are explicitly deferred. Do not add them unless the owner explicitly reopens the scope:

- AI-generated feedback on free-text reasoning (V2, needs backend)
- Persistent scores or user accounts (V2)
- Shareable recap card, multiplayer (V3)
- Creator/editor tooling for new rooms (V3)
- Native mobile app (not planned)
